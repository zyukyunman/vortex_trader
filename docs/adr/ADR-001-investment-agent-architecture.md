**相关：** vortex_common [ADR-001 部署架构](../../../vortex_common/docs/adr/ADR-001-deployment-architecture.md)、[design/01 端到端架构](../../design/01-端到端架构.md)、[design/04 研究院方法论](../../design/04-研究院方法论与自主研究闭环.md)、[design/05 L3 能力阶梯](../../design/05-L3自治交易能力阶梯.md)

## 背景（Context）

在 vortex_common 的部署架构里，`vortex_trader` 是**预留服务**（规范端口 8768）。本 ADR 定义它装什么：**一个投资 agent**——自主量化研究员/交易员，vortex 的服务 API 就是它的"手"。

**owner 诉求（已明确）：**
- 要一个投资 agent，**自己找合理的策略与因子**，调用 data/backtest/qmt 接口去**验证**。
- 重心是**回测验证**；验证好之后"策略即代码"，再去实盘。
- 更进一步：**agent 自己在环交易（L3）**——owner 倾向 L3，理由是"没太多时间维护策略代码"，并好奇 L3 的天花板。

**2026 技术现状（本仓调研结论，详见 design/04、design/05）：**
- **RD-Agent(Q) + Qlib** 已是**自治因子/模型研发**的成熟开源范式：五段式 LLM 闭环（Specification→Synthesis→Implementation→Validation→Analysis），持续自迭代、因子模型联合优化。即"agent 自己找因子/策略"已有成品。
- **但 LLM 在环实盘没有可信的长期盈利记录**：静态基准强 ≠ 实盘强（LiveTradeBench）；LLM 有"偷看未来"偏差（Look-Ahead-Bench）；交易成本常被低估；无风控的 agent 会垮。
- **2026 生产共识**：ML 做低延迟信号 + LLM 做策略级决策 + **确定性规则做硬风控**，且**人监督不可少**。

**关键约束（影响方案）：**
- **qlib 已被平台运行时 base 出局**（见 vortex_common ADR-001：「qlib 已出局 → 镜像跨架构」）。而本 agent 的研发内核需要 qlib/RD-Agent —— **重依赖与精简运行时基座存在张力，必须解耦**。
- 平台运行形态＝单镜像 + 多容器（一进程一容器），`vortexctl <svc>` 启动，各仓提供 `deploy/run.sh`。trader 需对齐这套（端口 8768、命名卷）。
- 数据双路径：vortex_data(Tushare) 与 qmt-bridge(xtquant) 都能取数；qlib 导出是 data→研究 的既有契约。

## 决策（Decision）

**核心原则一句话：agent 负责"找与决策"，确定性代码负责"关与限"；任何决策都必须可回放、可验证、可治理。**

**目标形态＝ L3 在环交易**，但以"**验证优先 + 确定性风控外壳 + 可回放策略产物**"为前提，按能力阶梯分级灰度（L3a→L3e，见 design/05）。"策略即代码"（L2）不是被放弃，而是被吸收为一种可选产物——agent 既能产出确定性策略代码，也能作为可回放的在环策略本身。

五条子决策：

1. **研发内核（让 agent 替你维护策略，而非你维护代码）**：以 **RD-Agent(Q) + Qlib** 作为自治因子/模型研发引擎（假设→实现→回测→反馈闭环），叠加公式型 alpha 挖掘（AlphaGen/AlphaForge 类）。agent 持续挖掘、回测、淘汰衰减 alpha——**owner 的"低维护"由此实现：维护的是 agent，不是一堆策略**。

2. **验证脊柱（不可妥协）**：所有候选必须过 **point-in-time 防偷看未来 + 真实成本/滑点 + 样本外/分段/分板 + A 股口径精撮合**。**双引擎**：Qlib 做因子/模型的快速筛选，**vortex_backtest 做 A 股真实撮合二次验证**（T+1/分板手数/涨跌停/费用——这是纯 Qlib 没有的、本平台的独特价值）。

3. **L3 决策层（可回放产物）**：agent 的决策策略**本身版本化、可回放**——喂同一 `snapshot_id`，agent 必须产出同一目标组合。采用记忆+反思架构（长短期记忆、按业绩斜率反思）、多 agent 分析师+多空辩论、选择性共识降不确定性。这把 L3 从"黑箱赌博"变成"一个可回测的特殊策略"。

4. **治理外壳（architectural determinism）**：agent **不直连券商**。任何决策经**确定性硬风控**（复用 vortex_qmt fail-closed 12 规则 + 三重门禁 + 全局总开关）→ paper→sim→小额→放量**灰度** → 全量审计（记录状态/prompt/工具调用/决策/血统）。决策权在 agent，否决权在确定性代码。

5. **运行与依赖解耦**：vortex_trader 作为部署架构里的服务（端口 8768、命名卷、`deploy/run.sh`）。**研究/agent 层（qlib + RD-Agent 等重依赖）打成独立镜像或 optional extra**，与精简运行时 base 解耦——既享用 RD-Agent/Qlib，又不违背"base 不带 qlib"的平台决策。

## 考虑过的方案（Options Considered）

### 方案 A：策略即代码（L2 为终态）
agent 自治研发产出**确定性策略代码**，代码跑回测与实盘，agent 退出执行环。

| 维度 | 评估 |
|------|------|
| 可治理/隔离 | **好**——实盘是确定性代码，行为可预期 |
| 低维护 | 好——agent 自动再生/迭代策略，owner 不手写 |
| 贴合 owner 诉求 | **中**——不满足"agent 自己在环交易"的好奇 |
| 成熟度 | 高——RD-Agent(Q) 即此范式 |

### 方案 B：L3 在环 + 验证脊柱 + 确定性外壳 + 可回放 ✅ 主
agent 每个调仓期自己决策目标组合，但经确定性风控、分级灰度、全程可回放。

| 维度 | 评估 |
|------|------|
| 可治理/隔离 | **好**——确定性外壳否决 + 可回放 + 审计 |
| 低维护 | 好——agent 既研发又决策 |
| 贴合 owner 诉求 | **高**——满足 L3 野心，且能安全探索天花板 |
| 成熟度 | 中——决策层前沿，靠外壳与灰度兜底 |

### 方案 C：纯 L3 黑箱
agent 直接下单，无可回放、无硬风控。

| 维度 | 评估 |
|------|------|
| 可治理 | **差**——不可复现、不可审计 |
| 安全 | **差**——证据显示无风控 agent 会垮，实盘不可接受 |
| 结论 | **否决** |

## 取舍分析（Trade-off Analysis）

owner 要"低维护 + L3 在环"，工程要"可治理/可验证"。看似冲突，**B 同时满足**：用「agent 维护研发」拿低维护，用「确定性风控外壳」拿可治理，用「可回放决策产物」拿可审计。代价是 L3 的治理/监控负担比 L2 重——用能力阶梯（design/05）分级灰度把风险摊薄、随证据逐级放开。

**"代码 vs 无代码 agent"是伪命题**：决定能否上实盘的不是策略长什么样，而是**能不能被回放验证、能不能被风控关住**。B 把这两条钉死，于是 L3 不再等于失控。

**qlib 出局 base 的张力**：用「研究层独立镜像/extra」解决——研发内核可以重，运行时基座保持精简，互不污染。

纯 A 被否的唯一理由是**不满足 owner 对 L3 的明确偏好**（A 在工程上其实最稳，作为 L3 不可用时的退路保留）。纯 C 因不可治理、实盘不可接受被否。

## 后果（Consequences）

**更容易：**
- agent 自动维护"策略研发"，owner 从手写/维护策略中解放。
- L3 可沿能力阶梯安全分级推进，而非"全有或全无"。
- 与部署架构对齐（trader 服务化、端口 8768、命名卷、run.sh）。

**更难 / 需注意：**
- **L3 治理/监控负担重于 L2**——需要 kill-switch、实时偏移监控、审计、回放复核。
- **qlib/RD-Agent 重依赖**需独立镜像/extra，构建与运行链路更复杂。
- **look-ahead 偏差必须严防**——LLM 在环回测尤甚；坚持 point-in-time，必要时匿名化/盲测。
- **实盘前必须多交易日模拟盘**稳定样本；真钱每一步仍需 owner 人确认（三重门禁）。
- alpha 会衰减——需持续再挖与淘汰（已纳入研发内核）。

**需要复审的点：**
- 若 L3e（无监督放量）长期无可信先例，维持"人在回路"为硬上限，不强行全自治。
- 研究内核选型（RD-Agent vs 自建）在 spike 后复审。
